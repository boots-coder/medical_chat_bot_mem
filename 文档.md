> 查阅 Mem0 文档我们发现，关于 `Graph Memory` 的**唯一**可配置项是：
>
> - **`Configurable Graph Threshold` (可配置图阈值)**
>
> 请注意，这个 `threshold` (阈值) **不是**用来定义 schema 的。
>
> - **它的作用**：是用来解决“实体冲突”（`Entity Conflict`）的。
> - **它的逻辑**：它决定了 Mem0 在多大程度上认为“Bob”和“Robert”是同一个人（即“相似度”要多高才能合并节点）。
> - **它不能做什么**：它不能定义“Bob”是一个 `Patient` 类型的节点，也不能定义 `Patient` 节点应该有 `P_` 前缀。
>
> 这进一步证明了 Mem0 的“自动”特性：它只关心“匹配”，而不关心“定义”。

## 概念

**session**

> 把“Session”想象成一次完整的“挂号问诊”：
>
> - **Session开始：** 你走进诊室，说：“医生，我不舒服。”（用户发起新对话）
> - **Session进行中：** 医生问你“哪里不舒服？”、“多久了？”...你一一回答。（多轮对话）
> - **Session结束（你问的关键点）：** 你问“如何定义结束？”
>   - **情况A（显式结束）：** 医生给了诊断和药方，你说了“谢谢医生，我明白了，再见”。（用户说了“谢谢”、“再见”、“没事了”）
>   - **情况B（超时结束）：** 医生给了药方，你拿着药方在原地坐着，10分钟没再说话（也没离开诊室）。医生会认为这次问诊“已结束”。（用户在一段时间内没有再发消息）
> - **总结存档（我们讨论的“存”）：** 在你离开诊室后（Session结束后），医生在你的“病历本”上写下摘要：“患者主诉头痛，诊断为偏头痛，已开具XX药。”（这就是我们要存入向量库的“摘要”）

定义**Session**结束

> 用户说谢谢 = 结束：这个逻辑（显式意图检测）是合理且必要的。
>
> 默认30mins结束 = 结束：这个逻辑非常不合理，风险很高。

- 什么时候开始走存储流程： 一次session结束后。 

**定义（最优子树）**：一个“最优子树”（或称“**连贯医疗主题块**”，Coherent Medical Topic Block）是指对话中一个片段**，它围绕**一个核心医疗意图/主题重构最小上下文

>  目的是，保证存储的“原子性” (Atomicity) 和“一致性” (Consistency)。

2 常规数据库 (SQL）

**作用：** 用来“精确查找”。

**存什么：** 存“事实”。例如：`user_id`、`session_id`、`start_time`。它用来回答：“找出张三的**所有**历史会话ID”。

3 向量数据库 (Vector DB)

**作用：** 用来“模糊查找”或“找相似”。

**存什么：** 存“经验”和“描述”。它用来回答：“帮我找找**所有类似‘头痛恶心’**的咨询”。

**`text_summary`** (连贯的文本摘要)：

- *“用户主诉头痛恶心2天，有高血压病史。机器人评估为偏头痛，建议休息并可服用布洛芬。”*

（可选）**`main_complaint`** (主诉)：

- *“头痛，伴随恶心，已持续2天”*

4 图数据库 (Graph DB)

`key_symptoms` (症状)

`medical_history` (病史)

`mentioned_drugs` (药物)

`suggested_diagnosis` (诊断)

以及它们和 `user_id` 的关系。

**点 (Nodes)：**

- `(User {id: "user-abc-789"})`
- `(Symptom {name: "头痛"})`
- `(Symptom {name: "恶心"})`
- `(Disease {name: "高血压"})`
- `(Diagnosis {name: "偏头痛"})`
- `(Drug {name: "布洛芬"})`

**边 (Edges / Relationships)：**

- `(User) -[HAS_SESSION]-> (Session {id: "session-xyz-123"})`
- `(Session) -[COMPLAINED_OF]-> (Symptom {name: "头痛"})`
- `(Session) -[COMPLAINED_OF]-> (Symptom {name: "恶心"})`
- `(User) -[HAS_HISTORY]-> (Disease {name: "高血压"})`
- `(Session) -[SUGGESTED_DIAGNOSIS]-> (Diagnosis {name: "偏头痛"})`
- `(Diagnosis) -[RECOMMEND_DRUG]-> (Drug {name: "布洛芬"})`



也就是说，到目前为止我们定义了什么是session，什么是session的结束，同时我们确定了session结束后，就要开始一次摘要总结。我们会针对session内的所有对话（我们认为这应该是一个问答格式的list）。

然后把这个list发送给一个llm api。然后api会返回我们一个固定格式的摘要内容。 然后根据摘要，我们直接进行拆分，分别存储到SQL ， VDB， GDB里。

现在我需要一个提示词，或者说一个实际的类，得到一个list（一次session的所有对话），返回三种格式的存储信息。 这个类要进行包括llm 调用的操作。

## 对话分析代码实现 -- 上下文管理

````python
from openai import OpenAI
from typing import List, Dict
import json


class TokenLimitExceeded(Exception):
    """对话token数超出限制异常"""
    pass


class DialogueAnalyzer:
    """对话分析器：将医疗对话转换为三层存储结构"""
    
    def __init__(
        self, 
        api_key: str,
        base_url: str = "https://openrouter.ai/api/v1",
        model: str = "openai/gpt-4o-mini",
        max_input_tokens: int = 4096
    ):
        self.client = OpenAI(base_url=base_url, api_key=api_key)
        self.model = model
        self.max_input_tokens = max_input_tokens
        
    def analyze_session(
        self, 
        dialogue_list: List[Dict[str, str]],
        session_id: str,
        user_id: str
    ) -> Dict:
        """
        分析对话session，返回三层存储结构
        
        Args:
            dialogue_list: [{"role": "user", "content": "..."}, {"role": "assistant", "content": "..."}]
            session_id: 会话ID
            user_id: 用户ID
            
        Returns:
            包含SQLite、VectorDB、GraphDB数据的字典
        """
        # TODO: 实现token计数逻辑
        # estimated_tokens = self._estimate_tokens(dialogue_list)
        # if estimated_tokens > self.max_input_tokens:
        #     raise TokenLimitExceeded(f"对话tokens({estimated_tokens})超出限制({self.max_input_tokens})")
        
        # 构建提示词
        system_prompt = self._build_system_prompt()
        user_prompt = self._build_user_prompt(dialogue_list, session_id, user_id)
        
        # 调用LLM
        completion = self.client.chat.completions.create(
            model=self.model,
            messages=[
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": user_prompt}
            ],
            response_format={"type": "json_object"},
            extra_headers={
                "HTTP-Referer": "https://medical-dialogue-system.com",
                "X-Title": "Medical Dialogue Analyzer"
            }
        )
        
        # 解析返回的JSON
        result = json.loads(completion.choices[0].message.content)
        return result
    
    def _build_system_prompt(self) -> str:
        """构建系统提示词"""
        return """你是一个专业的医疗对话分析助手，负责将医患对话转换为结构化的三层数据存储格式。

## 任务目标
分析医疗对话，提取信息并输出JSON格式，包含三个层次：
1. **SQLite（事实层）**：会话元数据（时间、轮次、主题）
2. **VectorDB（语义层）**：对话摘要和主诉的语义化表达
3. **GraphDB（关系层）**：医学实体及其关系的知识图谱

## 输出格式规范
```json
{
  "session_id": "会话ID",
  "user_id": "用户ID",
  "start_time": "ISO8601格式开始时间",
  "end_time": "ISO8601格式结束时间",
  "dialogue_rounds": 对话轮次数(整数),
  "session_topic": "一句话概括对话主题",
  
  "narrative_summary": "3-5句话完整叙述对话过程，包含主诉、评估、建议",
  "main_complaint_vectorized": "提取用户主诉的核心关键词组合",
  
  "knowledge_graph": {
    "entities": [
      {"id": "唯一标识", "type": "实体类型", "label": "实体名称"}
    ],
    "relationships": [
      {"subject": "主体实体ID", "predicate": "关系类型", "object": "客体实体ID"}
    ]
  }
}
```

## 实体类型规范
- **Patient**: 患者（使用user_id作为ID）
- **Symptom**: 症状（ID前缀：`S_`，例如 `S_Headache`）
- **Disease**: 疾病（ID前缀：`D_`，例如 `D_Hypertension`）
- **Diagnosis**: 诊断结论（ID前缀：`DG_`，例如 `DG_TensionHeadache`）
- **Drug**: 药物（ID前缀：`DR_`，例如 `DR_Ibuprofen`）
- **Examination**: 检查项目（ID前缀：`E_`，例如 `E_BloodPressure`）
- **Treatment**: 治疗方案（ID前缀：`T_`，例如 `T_BedRest`）

## 关系类型规范
- **HAS_SYMPTOM**: 患者有症状
- **HAS_HISTORY**: 患者有病史
- **MAY_CAUSE**: 疾病可能导致症状
- **IS_SUGGESTED_FOR**: 诊断建议给患者
- **RECOMMENDED_FOR**: 药物/治疗推荐给诊断
- **REQUIRES**: 诊断需要检查
- **INDICATES**: 检查结果指示疾病

## 注意事项
1. 实体ID使用英文，label使用中文
2. 时间戳使用ISO 8601格式（UTC）
3. 所有实体ID必须标准化（未来对接ICD-10、SNOMED CT等医学编码库）
4. 关系必须准确反映医学逻辑
5. 必须返回合法的JSON格式

请严格按照上述格式输出。"""

    def _build_user_prompt(
        self, 
        dialogue_list: List[Dict[str, str]],
        session_id: str,
        user_id: str
    ) -> str:
        """构建用户提示词"""
        # 格式化对话历史
        dialogue_text = ""
        for i, turn in enumerate(dialogue_list, 1):
            role = "用户" if turn["role"] == "user" else "助手"
            dialogue_text += f"[第{i}轮 - {role}]: {turn['content']}\n\n"
        
        return f"""请分析以下医疗对话，提取结构化信息。

                ## 会话信息
                - session_id: {session_id}
                - user_id: {user_id}

                ## 对话内容
                {dialogue_text}

                请按照系统提示中的JSON格式输出分析结果。"""
    
    def _estimate_tokens(self, dialogue_list: List[Dict[str, str]]) -> int:
        """
        估算对话的token数量
        
        TODO: 使用tiktoken库精确计数
        简单实现：中文1字≈1.5tokens，英文1词≈1.3tokens
        """
        total_chars = sum(len(turn["content"]) for turn in dialogue_list)
        estimated_tokens = int(total_chars * 1.5)  # 粗略估算
        return estimated_tokens
````

### 3 如果整个session内的对话超出了可执行的预期如何操作？ 

> 主要需要解决的是存储单元的可靠性。
>
> 1 很长， 并且都有用，一次提供了很多医疗有用的信息，需要精确的存储，这个时候可以使用聚类。
>
> 2 很长，并且大多数都没用， 如何找到有用的医疗信息。这个时候是大海捞针的场景。 
>
> 3 很短，可以直接调用 我们现有的代码，并且由守卫来进行过滤。

（核心挑战与亟待优化的目标）

- 方案的核心思想是 聚焦思路，focus关键信息， 重构上下文

- 我们直接针对5/10轮对话进行构建，然后聚合。 （好处是可以利用短期对话保留的摘要信息）

**“查” (Search) 的挑战：查询路由**

- **现状：** “存”（Write）是统一的（LLM -> 3 DBs）。
- **问题：** “查”（Read）是分散的。当一个新 Session 开始，机器人如何利用这三份记忆？
- **场景：** 用户说：“我上次说的那个头痛，现在感觉好多了。”
- **解决：** 需要一个**“查询路由” (Query Router)**。当新消息进来时：
  1. **短期记忆（滑动窗口）：** 首先一定在当前 Session 的滑动窗口中查找上下文。-- 短期记忆保证session内对话连贯性。
  2. **查询路由 (LLM)：** 如果遇到关键字匹配，触发“查询路由”。这个 Router (可以是另一个轻量级 LLM Prompt) 会决定：
     - “这是个新问题” -> (无需查)
     - “这是在问历史经验” -> (查 VectorDB)
     - “这是在问‘我’的病史” -> (查 GraphDB，`WHERE user_id = 'user_789'`)
     - “这是在问某个具体 Session” -> (查 SQLite)
  3. **结果融合：** 将查到的结果（例如，来自 VDB 和 GDB 的信息）融合后，注入到下一步的 Prompt 中。



## update 实现与思考

后面如何修改冲突的历史记录？

1. **“改” (Update) 的挑战：知识图谱的生命周期**

   - **问题：** 病人的情况是会变的。
   - **场景：** `Session 1` 存入了 `(user_789) -[HAS_SYMPTOM]-> (S_Headache)`。在 `Session 5`，用户说：“我头痛已经好了”。
   - dilemma： 是应该删除 `HAS_SYMPTOM` 这条边，还是增加一个 `(S_Headache) -[HAS_STATUS]-> (Resolved)` 节点？
   - 借用“时态图数据库”(Temporal Graph) 的概念。最简单的实现是：**在“关系”上增加属性**。
     - 修改 `relationships` 结构为： `{"subject": "user_789", "predicate": "HAS_SYMPTOM", "object": "S_Headache", "properties": {"session_id": "sess_1a2b3c", "status": "active"}}`
     - 当 `Session 5` 进来时，你可以查询到这条“活跃”的边，并在 `Session 5` 存储时将其更新为 `"status": "resolved"`。

## 总结

为什么我们不使用mem 0 ，而是自主开发一套框架？

- 存储时机与数据粒度的不同（“何时存”与“存什么”）；如果mem0 一样每句话进行add，blind。
- 对数据有**完全的控制权**
- 对信息提取（LLM）的绝对控制力（“存什么”） -- 后续如果想要规范化graphDB ,非常方便进行改进。

总体而言我们的架构是mem0 思想在医疗领域的定制化实例，对该双路（多路）rag思想进行了控制反转。
